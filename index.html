Switch from Shapefile

ESRI Shapefile is old file format for storing geospatial vector data. It has been around since early 1990s. It is still used as most common exchange vector data format.

The geospatial IT industry made a huge step since then, yet the Shapefile is still most common file format for sharing vector data.

Shapefile was great format in it's time and the fact, it's still used today, proofs, that it's design was truly eternal.

Yet we believe, the time of the Shapefile has passed and we, the Geospatial IT Industry shall replace it with more modern concepts. Shapefile is now outdated format and should be abandoned in the future for sharing of geospatial vector data.

The good side

Not all Shapefile features do make it bad (or unmodern) format. There is a couple of features, why Shapefile is still beating most of other formats:

  • It's open format. Even proprietary, the specification is open and implemented wildly.
  • Shapefile is by far mostly supported format. If every other format fails, Shapefile is always supported
  • For most of the usecases, it appears to be good enough.
  • It appears to be relatively efficient - the resulting file, even un-ziped - is relatively small, compared to some other, mostly text-based formats.

Shapefile is bad format

Why is Shapefile so bad? Here are several reasons, why the Shapefile is bad format and you should avoid it's usage:

  • It's multifile format. You can not share just one file, you always need to send at least 3, or zip them together
  • Attribute names are limited to 10 characters. If you need longer attribute names, you are lost.
  • Only 255 attributes. The DBF file does not allow to store more then 255 attribute fields.
  • Limitted data types. Data types are limited to float, integer, date and text with maximum 254 characters.
  • Unknown character set. There is now way how to specify character set used in the database.
  • It's limited to 2GB of file size. Even some tools are able to overpass this limit, they can never exceed 4GB of data.
  • No topology in the data. There is no way, how to describe topological relations in the format.
  • Single geometry type per file. There is no way, how to save mixed geometry features.
  • More complicated data structures are impossible to save. It's "flat table" format.

Multifile format

The Shapefile uses at least 3 files, but there are up to 10 other file extensions, which can be distributed along with the Shapefile. Nearly every geospatial software package is adding it's own extension to try to overcome some of the limit's Shapefile has.

This is highly unpractical for distribution of the data. User usually has to zip all the files into one archive - and unzip it on the other end of the distribution chain.

10 Characters attribute names

Attribute names are limited to 10 characters at maximum. Longer names are usually automatically shorten. This is limiting to some application and leads to bad data arrangement.

255 attribute fields

There can be only 255 attribute fields in the database file. For some application this is limiting especially in combination with the flat table structure.

Poor support for attribute data types

Float, integer, date and character string data types are supported. But floating point numbers are stored as text, there is no support for big integers (thus the format is not usable, you are having data with big integer identifiers, such as cadastral map) and the text is limited to 254 characters only.

There is no support for more advanced data fields, blobs etc.

Unknown character set

There is now way how to specify character set used in the database. Most of the applications are using the old Windows-* or ISO-* data encodings, but nowadays, we are using UTF-8. Still there is no way how to specify this in file header.

The support for Unicode characters is also very limited.

2GB Size limit

The size of both .shp and .dbf component files cannot exceed 2 GB. GDAL Shapefile driver overcomes this limit, but

The Shapefile format explicitly uses 32bit offsets and so cannot go over 8GB (it actually uses 32bit offsets to 16bit words), but the OGR shapefile implementation has a limitation to 4GB.

However, for compatibility with other software implementation, it is not recommended to use a file size over 2GB for both .SHP and .DBF files.

So 4GB is all you can have in single Shapefile. This sounds enough, but not for all cases.

Non-topological format

Shapefile is simple-feature format. There is no way, how to store more complex geometry relationships.

No mixed geometry

Each file can be only one of the supported geometry formats (Point, Line, Polygon and others). Mixed geometry features are not possible.

Flat data structure

The data structure is limited to flat tables with no hierarchies, relations or tree structure.

Alternatives

What are the alternatives for the Shapefile? To be honest, no alternative format did take over Shapefiles hegemony yet. Some of the formats nearly took over (KML, GML, GeoJSON), but their usage was limited for relatively narrow use cases only.

Although there are more then 80 used vector data formats out there, only few can be considered as the candidate for Shapefile replacement.

List of some Shapefile alternatives
  • OGC GeoPackage
  • OGC GML
  • ESRI GeoDatabase
  • SpatiaLite
  • CSV
  • OGC KML
  • GeoJSON

OGC GeoPackage

OGC GeoPackage is one of the most promising formats, designed for today's modern applications. GeoPackage is published as standard byOpen Geospatial Consoritum.

Features

  • SQLite as backend
  • File based, single file
  • Vectors, rasters
  • Official extensions
  • Supported in many software packages

Description

GeoPackage is an open, standards-based, platform-independent, portable, self-describing, compact format for transferring geospatial information.

The GeoPackage Encoding Standard describes a set of conventions for storing the following within an SQLite database:

  • vector features
  • tile matrix sets of imagery and raster maps at various scales
  • attributes (non-spatial data)
  • extensions

There are already published extensions for GeoPackage, who make this format even more powerful.

GeoPackage is now (2017) supported in most of the software packages.

We believe, what GeoPackage is the candidate for Shapefile replacement.

OGC GML

Another OGC Standard.

Features

  • XML Based
  • Only vectors
  • Hierarchies
  • Thanks to INSPIRE, supported in most of software package

Description

GML was picked as the main distribution vector data format the European INSPIRE initiative. It's very complex format, but it's usage is hard in daily praxis. One of the issues is, that in order to being able to process the data file, it has to be loaded into computer memory, which makes it very difficult to be used for large vector data sets.

GML is currently often use for open data data sets, since it's technology neutral and supported OGC Standard.

ESRI GeoDatabase

At its most basic level, an ArcGIS geodatabase is a collection of geographic datasets of various types held in a common file system folder, a Microsoft Access database, or a multiuser relational DBMS (such as Oracle, Microsoft SQL Server, PostgreSQL, Informix, or IBM DB2).

Features

  • native data structure for ArcGIS
  • file-based (or database based)
  • complex data models
  • proprietary, closed format

Description

GeoDatabase is very often used in the ArcGIS environment as the main exchange data format. It's features are very complex and advanced.

On the other hand, it's proprietary closed format, which is used exclusively in the environment of ESRI products, not implemented in other software packages.

SpatiaLite

SpatiaLite is popular database, file based data storage.

Features

  • File based
  • SQL database
  • OGC Simple feature

Description

SpatiaLite is an open source library intended to extend the SQLite core to support fully fledged Spatial SQL capabilities. SQLite is intrinsically simple and lightweight:

  • a single lightweight library implementing the full SQL engine
  • standard SQL implementation: almost complete SQL-92
  • a whole database simply corresponds to a single monolithic file (no size limits)
  • any DB-file can be safely exchanged across different platforms, because the internal architecture is universally portable

    • To us, SpatiaLite seems to be the second best option after the GeoPackage possibility. They do build on top of the same technology, which is SQLite.

    Compared to GeoPackage, it lacks support for extensions and support for raster data - that is certainly not a must-have feature, but it's good, if we take this into consideration.

CSV

Some people tend to use comma separated files for storing of vector data.

Features

  • Simple

Description

Among non-GeoSpatial people, the CSV is very popular, but for most of GeoSpatial applications it's unusable format.

At least two reasons for not using CSV as Shapefile replacement: It's not standardized (there are many dialects out there), support for non-point geospatial data is complicated.

OGC KML

Originally provided by Google OGC KML used to be very popular vector data format.

Features

  • file-based
  • XML
  • Combines geometry along with cartography
  • Supports just WGS-84 coordinate system

Description

Some years back, the KML format was very popular, but the geospatial community hit on it's limits. Since XML based, it's not suitable for bigger data data sets. It combines cartography along with the data geometry in one file, what does not seem to be good sollution. And it officially supports only WGS-84 coordinate reference system.

GeoJSON

GeoJSON is community format, based on the popular JSON data exchange format.

Features

  • JSON format
  • File based
  • Can handle complex data
  • File size groves fast

Description

GeoJSON is very simple, human-readable, text-based format. Although it was design to support WGS-84 only, it's possible to use it for other coordinate reference systems too. It can handle complex vector data features, build complex hierarchical data models.

The problem with GeoJSON is, that the file grows very fast with amount of coordinate pairs needed to be stored. Indexing and other advanced features is not possible too.
